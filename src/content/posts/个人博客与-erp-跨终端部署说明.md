我最近把个人博客发布到了 GitHub Pages。它有一个很直观的效果：只要打开网址，手机、平板、其他电脑都能看到同一个博客。

这件事很容易让人自然联想到另一个问题：ERP 系统能不能也这样部署？

答案是：可以拥有相同的访问体验，但不能使用完全相同的部署方式。

## 博客为什么可以直接打开

个人博客使用 Astro 构建。它最终发布到 GitHub Pages 的不是一套正在运行的程序，而是一组静态文件：

```text
HTML
CSS
JavaScript
图片与静态资源
```

更新博客时，本地修改会先上传到 GitHub 仓库。随后 GitHub Actions 自动执行构建，把源代码转换成可发布的静态网页，再交给 GitHub Pages 托管。

整个过程可以理解为：

```text
本地写内容
  → 上传 GitHub
  → 自动构建
  → 发布到 GitHub Pages
  → 任何终端通过网址访问
```

这种网站不需要数据库，也不需要长期运行后端服务。访问者看到的是已经生成好的页面，所以 GitHub Pages 很适合博客、作品集、文档站和课程展示页。

## ERP 为什么不一样

ERP 系统表面上也是一个网页，但它背后有更多运行时能力：

- 用户登录和会话管理。
- 订单新增、编辑、删除和查询。
- Excel 导入与导出。
- 数据库备份和恢复。
- SQLite 数据库持续保存业务数据。
- 小程序订单同步接口。

这些能力都需要一台服务器持续运行 Python 程序，并保存不断变化的数据。

GitHub Pages 只能托管静态文件，不能持续运行 Python，也不能保存 ERP 数据库。因此，GitHub 可以继续保存 ERP 的代码，但真正的 ERP 服务必须部署在服务器、云主机或应用托管平台上。

## 要实现的真实体验

从用户角度看，ERP 最终也可以像博客一样简单：

```text
打开一个 HTTPS 地址
  → 进入 ERP 登录页
  → 登录后管理订单、客户、供应商和财务数据
  → 多个终端看到同一份业务数据
```

区别在于，博客的网址背后是静态托管服务，而 ERP 的网址背后应该是一个长期运行的后端系统。

## 三种部署路线

第一种是局域网部署。把 ERP 放在公司内部一台长期运行的电脑或 Windows Server 上，员工通过 `http://192.168.x.x` 这类局域网地址访问。它成本低，适合试运行，但离开公司网络后通常无法使用。

第二种是云服务器部署。购买一台云服务器，配置域名、HTTPS、Nginx、Python 服务和数据库备份。它更适合正式业务，因为访问稳定，数据也更容易做权限、备份和恢复管理。

第三种是应用托管平台。比如把 ERP 代码放在私有 GitHub 仓库，再部署到支持 Python 服务的平台，并挂载持久磁盘保存 SQLite 数据库。它上线快，但要特别注意免费服务的文件系统通常不是持久存储。

## ERP 上线前必须加固什么

ERP 不建议直接暴露到公网。正式部署前，至少要完成这些工作：

1. 修改所有初始密码。
2. 使用私有仓库保存 ERP 代码。
3. 不上传 `erp.db`、备份文件、密钥和真实客户资料。
4. 启用 HTTPS。
5. 为登录 Cookie 增加生产环境安全配置。
6. 为新增、修改、删除等操作补充 CSRF 防护。
7. 把数据库放在持久磁盘。
8. 每天自动备份，并复制到服务器之外。
9. 限制数据库恢复、批量删除和用户管理等高风险操作。
10. 做一次完整的上线测试，包括断电恢复和备份恢复。

这些听起来像额外工作，但对 ERP 来说不是装饰，而是系统能不能长期被信任的基础。

## 我会怎么推进

最稳妥的顺序不是一上来就公网开放，而是分三步：

```text
公司内部试运行
  → 安全与备份加固
  → 云服务器或持久化平台部署
```

内部试运行阶段可以先验证真实业务流程：订单录入是否顺手、财务字段是否够用、导入导出是否可靠、权限是否符合团队分工。

等流程稳定后，再处理公网访问、HTTPS、域名、备份、日志和监控。这样上线不是一次冒险，而是把已经跑通的系统搬到更稳定的位置。

## 结论

博客能直接跨终端打开，是因为它是静态网站，适合 GitHub Pages。

ERP 也能做到跨终端访问，但它是业务系统，需要服务器、后端服务和持久化数据库。

两者最终都可以通过一个网址访问，只是背后的基础设施不同：博客发布的是内容，ERP 运行的是业务。
