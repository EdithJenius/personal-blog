多 Agent 听起来像是“多安排几个角色”，但系统一旦进入真实任务，问题很快会落到三个地方：谁负责、信息放在哪里、失败后怎样继续。

## 角色不等于边界

给 Agent 起名字很容易，定义它能读什么、能写什么、何时结束才是关键。一个好边界应该让交接物明确，而不是依赖下一位 Agent 猜测上下文。

## 状态必须外显

长期任务不能只依赖对话历史。计划、决策、输出和未解决问题都需要进入稳定的文件或数据库结构。这样系统才具备恢复能力，也方便人类随时接管。

## 交付优先于讨论

每个 Agent 都应该面向一个可检查的产出：文档、数据、代码、报告或决策记录。没有交付标准的协作，最终只会制造更多信息。

多 Agent 的价值并不是模仿一个热闹团队，而是把复杂工作拆成可追踪、可复用、可验证的生产过程。
